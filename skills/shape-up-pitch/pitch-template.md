## Problem

*What's broken and why it matters. Specific, falsifiable, connected to
impact: who is hurt, how often, what it costs. Only evidence the writer
provided — uncertainty stays visible ("support estimates…, unverified").
"X is bad" or "users don't like it" does not pass.*

## Appetite

*How much time this is worth — a budget set by the writer, not an estimate
from engineering. State what happens at the boundary: scope gets cut, time
does not get extended. If the appetite was converted from an estimate,
say so and confirm with the writer.*

## Solution

*A direction, not a spec. Enough shape to be buildable; enough room for team
judgment. Fat-marker level: the key elements and how they connect —
breadboard or rough sketch words, no widget choices, no component names, no
field-level detail. If you can hand it to a designer as a finished UI
instruction, the altitude is wrong.*

## Rabbit Holes

*Known risks that could derail the cycle — named concretely so the team can
avoid them, each with the boundary decision that patches it ("decide X in
week one", "fallback is Y"). Technical unknowns, tempting over-builds,
ambiguities that eat days.*

## No-Gos

*Explicit scope exclusions decided before work begins. What is out, even if
it comes up mid-cycle. If this section is empty, the scope hasn't really
been decided — write "None identified yet" and add an Open Question.*

## Open Questions

*Everything that must be answered before the cycle starts: decisions the
writer left open (verbatim, not silently resolved), facts the pitch needs
but doesn't have, recommendations awaiting the writer's confirmation.
Any unresolved item here ⇒ Status is `shaping`, not `ready to bet`.*
